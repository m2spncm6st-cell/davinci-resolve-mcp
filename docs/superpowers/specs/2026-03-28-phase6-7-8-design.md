# Design: Phase 6–8 — Audio, Transitions, Marker-Editing

**Datum:** 2026-03-28
**Projekt:** davinci-resolve-mcp
**Ziel:** Professionelles Video-Editing per Prompt — breite Basis für beliebige Workflows

---

## Kontext

v1.1.0 hat 10 Tools und 75+ Actions. Die nächsten drei Phasen erweitern bestehende Tools gezielt um Audio-Kontrolle, Transitions und Marker-basiertes Editing. Jede Phase ist ein eigenständiger Schritt mit eigenem Tag, Tests und Commit.

Strategie: sequenziell (Audio → Transitions → Marker-Editing), vom Einfachsten zum Komplexesten.

---

## Phase 6 — Audio/Fairlight (v1.2.0)

### Scope

Erweiterung des bestehenden `fairlight` Tools um 6 neue Actions.

### Actions

| Action | Parameter | Beschreibung |
|--------|-----------|--------------|
| `get_volume` | `track_index` | Aktuelle Track-Lautstärke lesen |
| `set_volume` | `track_index`, `volume` (float, dB) | Track-Lautstärke setzen |
| `set_mute` | `track_index`, `muted` (bool) | Track muten oder unmuten |
| `set_pan` | `track_index`, `pan` (float, -1.0 bis 1.0) | Panorama setzen |
| `fade_in` | `track_index`, `item_index`, `duration` (int, frames) | Fade-In auf einem Clip setzen |
| `fade_out` | `track_index`, `item_index`, `duration` (int, frames) | Fade-Out auf einem Clip setzen |

### Architektur

- `fairlight(action, track_index, ...)` — bestehende Signatur um `item_index`, `volume`, `muted`, `pan`, `duration` erweitern
- `set_volume`/`set_mute`/`set_pan`: Timeline-Level via `tl` Objekt (Track-Properties)
- `fade_in`/`fade_out`: Item-Level via `TimelineItem.SetProperty("FadeInDuration"|"FadeOutDuration", frames)`
- Alle neuen Parameter als `| None = None` optional

### API-Realität

Track-Level-Volume/Mute/Pan sind in der Resolve Scripting API nicht über dedizierte Track-Methoden erreichbar — sie laufen über `SetTrackVolume`, `MuteAudioTrack` o.ä., deren Verfügbarkeit versionsabhängig ist. Wo die API fehlt: `_err` mit klarer Meldung, kein Absturz. Fades (`FadeInDuration`, `FadeOutDuration`) sind über `TimelineItem.SetProperty` zuverlässig verfügbar.

### Tests

6 neue Tests (einer pro Action). Alle 29 bestehenden Tests bleiben grün.

### Tag

`v1.2.0`

## Phase 7 — Transitions (v1.3.0)

### Scope

Neues Tool `transition` — eigenständig, weil Transitions konzeptuell unabhängig von Timeline-Items und Fairlight sind.

### Actions

| Action | Parameter | Beschreibung |
|--------|-----------|--------------|
| `list_types` | — | Verfügbare Transition-Typen auflisten |
| `get` | `track_type`, `track_index`, `item_index` | Aktuelle Transition an einem Clip lesen |
| `add` | `track_type`, `track_index`, `item_index`, `transition_type`, `duration` (frames) | Übergang zwischen Clip[item_index] und Clip[item_index+1] einfügen |
| `remove` | `track_type`, `track_index`, `item_index` | Übergang an einem Clip entfernen |

### Architektur

- Neues `@mcp.tool()` `transition(action, track_type, track_index, item_index, transition_type, duration)`
- `list_types`: Hardcodierte Liste (`"Cut"`, `"Dissolve"`, `"DipToColor"`, `"Wipe"`) — die API kennt keine dynamische Liste
- `add`: `TimelineItem.AddTransition(transitionType, duration)` — nur auf Edit-Page-Clips
- `remove`: `TimelineItem.DeleteTransition()` falls verfügbar, sonst `_err`
- `get`: `TimelineItem.GetTransition()` falls verfügbar

### API-Realität

`AddTransition` ist Resolve-versions-abhängig. Wo nicht verfügbar: `_err` mit Hinweis auf Resolve-Version. Das Tool wird in `server.py` registriert und in Tests mit Mocks abgedeckt.

### Tests

4 neue Tests. Bestehende Tests bleiben grün.

### Tag

`v1.3.0`

## Phase 8 — Marker-basiertes Editing (v1.4.0)

### Scope

Erweiterung des bestehenden `timeline` Tools um 4 neue Actions. Marker sind bereits in `timeline` implementiert (`get_markers`, `add_marker`, `delete_markers`) — neue Actions bauen darauf auf.

### Actions

| Action | Parameter | Beschreibung |
|--------|-----------|--------------|
| `get_marker_clips` | `color` (optional) | Clips zurückgeben, die mit einem Marker (optional: bestimmte Farbe) überlappen |
| `split_at_markers` | `color` (optional) | Timeline an allen Markern (oder Farb-Markern) aufsplitten |
| `delete_between_markers` | `marker_a_frame`, `marker_b_frame` | Alle Clips im Bereich zwischen zwei Marker-Frames löschen |
| `rename_clips_from_markers` | `color` (optional) | Clips nach nächstgelegenem Marker benennen |

### Architektur

- Alle Actions werden in den bestehenden `timeline(action, ...)` Block integriert
- Bestehende Parameter-Signatur unverändert — keine Breaking Changes
- `split_at_markers`: Iteriert `GetMarkers()`, ruft `SplitClip(item, frame)` pro Marker; ein fehlgeschlagener Split unterbricht nicht die restlichen (robust per try/except pro Split)
- `delete_between_markers`: Nutzt `GetMarkers()` → Bereichslogik → bestehende `DeleteClips()`-Implementierung
- `get_marker_clips`: Reine Logik — Marker-Frames mit Item-Ranges abgleichen via `GetItemListInTrack`
- `rename_clips_from_markers`: `SetClipProperty("Clip Name", marker_name)` auf dem nächstgelegenen Clip

### API-Realität

`SplitClip` ist in neueren Resolve-Versionen (≥ 18) verfügbar. Bei fehlendem `SplitClip`: `_err` pro betroffenem Marker mit Zähler wie viele erfolgreich waren. `delete_between_markers` und `get_marker_clips` sind vollständig aus bestehenden API-Calls zusammensetzbar.

### Tests

4 neue Tests. Bestehende Tests bleiben grün.

### Tag

`v1.4.0`

- Dynamische Transition-Liste (API gibt diese nicht aus)
