# Design: v1.9.0 — Timeline Trim, Multi-Clip-Append, Global MCP

**Datum:** 2026-03-30
**Status:** Approved
**Version:** v1.9.0

## Übersicht

Drei unabhängige Erweiterungen, die aus dem Sizilien-Erinnerungsvideo-Projekt und alltäglichen Workflow-Lücken entstanden sind:

1. **`timeline_item(action="trim")`** — In/Out-Punkt eines bereits auf der Timeline liegenden Clips setzen
2. **`media_pool(action="append_to_timeline")` Multi-Clip-Trim** — mehrere Clips mit je eigenem Trim in einem Call
3. **MCP User-Scope** — Server global in jedem Claude Code Verzeichnis verfügbar

---

## Feature A — `timeline_item(action="trim")`

### Problem
Clips die bereits auf der Timeline liegen können aktuell nicht getrimmt werden. Der einzige Workaround ist Delete + Neu-Einfügen via `append_to_timeline`.

### Lösung
Neue Action `trim` im bestehenden `timeline_item` Tool. Nutzt die Resolve API-Methoden `SetLeftOffset()` / `SetRightOffset()`.

### Neue Parameter (zu bestehenden hinzugefügt)
```
start_frame: int | None   # Absoluter Source-In-Frame (bei nativer Clip-fps)
end_frame: int | None     # Absoluter Source-Out-Frame (bei nativer Clip-fps)
start_s: float | None     # Source-In in Sekunden
end_s: float | None       # Source-Out in Sekunden
fps: float | None         # Clip-fps für Sekunden-Berechnung (auto-detect wenn nicht gesetzt)
```

### Identifizierung des Clips
Wie alle `timeline_item`-Actions: `track_type` (default "video"), `track_index` (default 1), `item_index` (1-basiert).

### Logik
1. Clip holen per `_get_item()` (bestehender Helper)
2. fps auto-detect: `item.GetSourceItem().GetClipProperty("FPS")` → float, fallback 25.0
3. Sekunden → Frames konvertieren:
   - `start_s` → `start_frame = int(start_s * fps)`
   - `end_s` → `end_frame = int(end_s * fps)`
4. `start_frame` → `item.SetLeftOffset(start_frame)`
5. `end_frame` → `item.SetRightOffset(total_source_frames - end_frame - 1)`
   - `total_source_frames` via `item.GetSourceItem().GetClipProperty("Frames")`
6. Response: `left_offset`, `right_offset`, `duration_frames`, `fps_used`

### Semantik
- `start_frame` / `start_s`: absoluter In-Punkt im Source-Clip (nicht relativ zum aktuellen Trim)
- `end_frame` / `end_s`: absoluter Out-Punkt im Source-Clip
- Beide optional: nur `start_frame` → nur linke Seite trimmen; nur `end_frame` → nur rechte Seite

### Fehlerbehandlung
- Kein `start_frame` und kein `end_frame` und kein `start_s` und kein `end_s` → `_err("Mindestens start_frame/start_s oder end_frame/end_s angeben")`
- `SetLeftOffset`/`SetRightOffset` returns False → `_err("Trim fehlgeschlagen — Offset außerhalb des Source-Clips?")`

## Feature B — Multi-Clip-Append mit Trim

### Problem
`media_pool(action="append_to_timeline")` unterstützt Trim nur für einen Clip pro Call. Beim Aufbau einer Timeline aus mehreren Clips sind N separate Calls nötig.

### Lösung
Neuer optionaler Parameter `clip_specs: str | None` als JSON-Array. Rückwärts-kompatibel: wenn nicht gesetzt, altes Verhalten.

### Parameter
```
clip_specs: str | None   # JSON-Array mit per-Clip-Daten (neu)
```

Alle bisherigen Parameter (`start_frame`, `end_frame`, `start_s`, `duration_s`, `fps`, `file_paths`) bleiben unverändert.

### JSON-Format
```json
[
  {"name": "clip1.mp4", "start_s": 10, "duration_s": 30},
  {"name": "clip2.mp4", "start_frame": 0, "end_frame": 249},
  {"name": "clip3.mp4", "start_s": 5, "end_frame": 500, "fps": 100},
  {"name": "clip4.mp4"}
]
```

Unterstützte Felder pro Clip:
- `name` (required): Clip-Name im aktuellen Media Pool Folder
- `start_frame`, `end_frame`: Source-Frames bei nativer fps
- `start_s`, `duration_s`: Sekunden-basiert (alternativ zu Frames)
- `fps`: Override für native fps (sonst auto-detect)

Mischung von Frames und Sekunden pro Clip erlaubt (wie beim Single-Clip-Trim).

### Logik
1. `clip_specs` parsen: `json.loads(clip_specs)`
2. Alle Clips aus aktuellem Folder laden
3. Für jeden Eintrag: Clip per Name suchen, Frames/Sekunden konvertieren (gleiche Logik wie Single-Clip)
4. Clip-Info-Dict aufbauen: `{"mediaPoolItem": clip, "startFrame": sf, "endFrame": ef}`
5. **Einen** `mp.AppendToTimeline([...])` Call mit allen Clips → Resolve fügt sie in Reihenfolge ein
6. Response: `appended: N`, `clip_names: [...]`, pro Clip `start_frame`/`end_frame`/`fps_used`

### Fehlerbehandlung
- JSON nicht parsebar → `_err("clip_specs ist kein gültiges JSON: ...")`
- Clip-Name nicht gefunden → `_err("Clip 'X' nicht gefunden. Verfügbar: [...]")`
- `AppendToTimeline` schlägt fehl → `_err("AppendToTimeline fehlgeschlagen")`

### Rückwärts-Kompatibilität
- `clip_specs` nicht gesetzt → bisheriger Code-Pfad unverändert
- Bisherige `file_paths`/`start_frame`/`end_frame`/`start_s`/`duration_s` Parameter weiterhin gültig

## Feature C — MCP User-Scope (Global)

### Problem
`.mcp.json` liegt in `~/Desktop/MCP Da Vinci/` — der MCP-Server ist nur verfügbar wenn Claude Code aus diesem Verzeichnis geöffnet wird.

### Lösung
Einmaliger `claude mcp add --scope user` Befehl. Speichert Konfiguration in `~/.claude/settings.json` → Server in jedem Verzeichnis verfügbar.

### Befehl
```bash
claude mcp add davinci-resolve --scope user \
  -e PYTHONPATH="/Library/Application Support/Blackmagic Design/DaVinci Resolve/Developer/Scripting/Modules" \
  -e RESOLVE_SCRIPT_API="/Library/Application Support/Blackmagic Design/DaVinci Resolve/Developer/Scripting" \
  -e RESOLVE_SCRIPT_LIB="/Applications/DaVinci Resolve/DaVinci Resolve.app/Contents/Libraries/Fusion/fusionscript.so" \
  -- /Users/Alex/.pyenv/versions/3.12.11/bin/python3 /Users/Alex/Projects/davinci-resolve-mcp/src/server.py
```

### Kein Code-Change nötig
`server.py` bleibt unverändert. Nur:
1. Befehl ausführen
2. CLAUDE.md mit neuem Registrierungsbefehl aktualisieren (User-Scope statt Project-Scope)
3. Bestehende `.mcp.json` in `~/Desktop/MCP Da Vinci/` kann bleiben oder entfernt werden
